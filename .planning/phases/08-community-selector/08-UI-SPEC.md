## Visual Patterns

### Community Cards ("Your Communities" section)

```
bg-card rounded-3xl border border-border/50 p-4
flex flex-col gap-2
active:scale-[0.98] transition-transform
```

Card anatomy (top to bottom):
1. Top row: community name (`font-heading font-bold text-base`) + role badge (`text-[10px] font-medium px-2 py-0.5 rounded-full`)
2. Middle row: member count (`text-sm text-muted-foreground`) + last activity (`text-xs text-muted-foreground`)
3. Bottom row: "Go to community" button (primary, full-width on mobile)

Role badge colors:
- admin: `bg-primary/10 text-primary`
- coach: `bg-secondary/30 text-secondary-foreground`
- client: `bg-muted text-muted-foreground`

### Community Browse Cards

```
bg-card rounded-3xl border border-border/50 p-4
flex flex-col gap-3
```

Card anatomy:
1. Community name (`font-heading font-bold text-base`)
2. Description (`text-sm text-muted-foreground line-clamp-2`) — omit row if empty
3. Member count (`text-xs text-muted-foreground`)
4. "Request to Join" button (primary, full-width)

After request sent — inline state change (no remount):
- Button: disabled, `bg-orange-500 text-white cursor-not-allowed`, label "Pending Approval"
- "Cancel request" link appears below button: `text-sm text-orange-500 underline text-center`

### "/communities" Page Layout

```
px-5 pt-16 pb-24   ← top accounts for logout button; bottom clears nav
```

Section order:
1. Page heading: `font-heading font-bold text-2xl mb-6` — "Communities"
2. "Your Communities" section — visible when user has approved memberships
3. "Pending" section — visible when user has pending join requests (shows pending cards, not full community cards)
4. "Browse Communities" section — visible when browse list is non-empty
5. Each section header: `font-heading font-bold text-base mb-3` with Roland Garros orange count badge for Pending section

Admin sees "+ Create" card as last item in "Your Communities" grid:
- Dashed border: `border-2 border-dashed border-border rounded-3xl p-4`
- Center content: `Plus` icon (w-8 h-8 text-muted-foreground) + "Create Community" label
- On click: opens CreateCommunityDialog

Card grid: `grid grid-cols-1 gap-4` (single column, mobile-first)

Trigger: Current community name + `ChevronDown` icon

Dropdown items:

- "Browse communities" link: `text-sm text-muted-foreground`, navigates to `/communities`

## Interaction Contracts

### Join Request Flow

1. User clicks "Request to Join" on browse card
2. Button enters loading state: `disabled`, spinner replaces icon (`Loader2 animate-spin w-4 h-4`)
3. On success: button transitions to "Pending Approval" (orange, disabled) + "Cancel request" link appears
4. Toast fires: "Request sent! You'll be notified when approved."
5. Card moves to "Pending" section on next render (or optimistic hide from Browse)

Implementation: `useTransition` + optimistic state per card — matches existing `InvitationManager` pattern.

### Community Switcher Navigation

1. User taps current community name in top bar
2. Dropdown opens below with animation: `opacity-0 → opacity-100, scale-95 → scale-100, duration-150`
3. User selects another community → navigate to `/c/[other-slug]/[role-home-for-that-community]`
4. User selects "Browse communities" → navigate to `/communities`
5. Clicking outside or pressing Escape closes dropdown

### Approve/Reject Request (Coach/Admin)

Approve:
1. "Approve" button: loading state with spinner
2. On success: row removed from pending section, count badge decrements
3. Toast: "Approved. [Name] is now a member." (success)

Reject:
1. "Reject" button clicked → RejectRequestDialog opens
2. Confirm "Reject Request" → loading state on confirm button
3. On success: row removed from pending section, count badge decrements
4. Toast: "Request rejected." (neutral)
5. Dialog closes

### Notification for Approved/Rejected User

- Approved user: in-app notification (Phase 5 system) with `join_approved` type
- Rejected user: in-app notification with `join_rejected` type
- No notification sent to coach/admin when someone requests — roster badge (D-41, D-46) is sufficient

### Route Skeleton Loading

Every route under `/c/[slug]/` shows a skeleton `loading.tsx` while the community layout resolves slug + membership. Skeletons use `Skeleton` component, 3-4 placeholder elements, no spinner.

## Accessibility

- Community cards: keyboard focusable, `aria-label="Go to [Community Name]"` on card link
- Role badge: `aria-label="Your role: [role]"` 
- Pending count badge on Clients tab: `aria-label="[N] pending join request[s]"` (matches existing notification badge pattern)
- Reject dialog: `DialogTitle` and `DialogDescription` always present (matches existing Dialog usage in AppNav)
- Community switcher: `aria-expanded`, `aria-haspopup="listbox"` on trigger; `role="listbox"` on dropdown
- Empty states: announced via `role="status"` wrapper

Create `supabase/migrations/00006_notifications_schema.sql` following the established migration pattern from 00005.

The migration must contain:

1. **Table definition:**
```sql
create table public.notifications (
  id uuid primary key default gen_random_uuid(),
  community_id uuid references public.communities(id) on delete cascade not null,
  member_id uuid references public.community_members(id) on delete cascade not null,
  notification_type text not null check (
    notification_type in ('session_reminder', 'announcement', 'rsvp_confirmed', 'waitlist_promoted')
  ),
  title text not null,
  body text not null,
  metadata jsonb not null default '{}',
  read_at timestamptz,
  created_at timestamptz default now()
);
```

2. **Enable RLS:**
```sql
alter table public.notifications enable row level security;
```

3. **Unique index for cron idempotency (session reminders):**
```sql
create unique index notifications_session_reminder_unique
  on public.notifications(member_id, notification_type, (metadata->>'session_id'))
  where notification_type = 'session_reminder';
```

4. **Performance indexes:**
```sql
create index idx_notifications_member_created
  on public.notifications(member_id, created_at desc);
create index idx_notifications_community
  on public.notifications(community_id);
create index idx_notifications_unread
  on public.notifications(member_id) where read_at is null;
```

5. **RLS policies:**

SELECT â€” members read own notifications only:
```sql
create policy "Members read own notifications"
  on public.notifications for select
  using (member_id = (
    select id from public.community_members
    where user_id = auth.uid() limit 1
  ));
```

UPDATE â€” members mark own as read only:
```sql
create policy "Members mark own notifications read"
  on public.notifications for update
  using (member_id = (
    select id from public.community_members
    where user_id = auth.uid() limit 1
  ))
  with check (member_id = (
    select id from public.community_members
    where user_id = auth.uid() limit 1
  ));
```

INSERT â€” allow inserts from authenticated users in the same community (needed for server actions that notify other members in the same community). The server action already validates role and community membership before inserting:
```sql
create policy "Authenticated users can insert notifications in their community"
  on public.notifications for insert
  with check (
    community_id = ((select auth.jwt()) ->> 'community_id')::uuid
  );
```

6. **Add table to Supabase Realtime publication (CRITICAL for D-02 live updates):**
```sql
alter publication supabase_realtime add table public.notifications;
```

**`markAllNotificationsRead()`:**
1. Get authenticated user via `supabase.auth.getUser()`
2. Look up `community_members.id` via `user_id = user.id`
3. Update all notifications where `member_id = member.id` and `read_at is null`, setting `read_at` to `new Date().toISOString()`
4. Call `revalidatePath('/notifications')`
5. Return `{ success: true }` or `{ success: false, error }` on failure

**`markNotificationRead(notificationId: string)`:**
1. Get authenticated user via `supabase.auth.getUser()`
2. Look up `community_members.id` via `user_id = user.id`
3. Update the single notification matching `id = notificationId` AND `member_id = member.id` (security: prevents marking other members' notifications), setting `read_at` to `new Date().toISOString()`
4. Call `revalidatePath('/notifications')`
5. Return `{ success: true }` or `{ success: false, error }` on failure

Both actions must:
- Return `{ success: false, error: 'Not authenticated' }` if no user
- Return `{ success: false, error: 'Member record not found' }` if no member
- Use the `member_id` filter on every update query (T-05 security: prevents horizontal privilege escalation)
